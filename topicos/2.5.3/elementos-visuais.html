Exemplos para a diretriz 2.5.3 - Rótulo no nome acessível

Apresentamos alguns elementos visuais que podem indicar ações na navegação e com boas práticas tornarão a página acessível independente da tecnologia utilizada no desenvolvimento.

Exemplo: Botão

No exemplo abaixo temos 5 botões vermelhos, um ao lado do outro, estilizados com o MATERIALIZECSS.

save cloud salvar cloudsalvar

Sobre o primeiro botão:

É um botão redondo com o ícone de um disquete, com a ideia de indicar um botão para salvar. Porém o disquete fez parte de uma época passada que nem todos viveram e atualmente os mais jovens podem ter dificuldade de relacionar o elemento com o verdadeiro objetivo, se for mantido apenas o ícone no botão sem qualquer outro tipo de informação.

Sobre o segundo botão:

É um botão redondo com o ícone de uma nuvem, com a ideia de indicar de salvamento, como o anterior. Algumas vezes observamos a substituição do ícone do disquete para o ícone de nuvem para representar o salvamento na cloud. Porém vale lembrar que nem todos os usuários já têm ideia de cloud e podem não conseguir relacionar o ícone com o verdadeiro significado se o botão for mantido apenas com o ícone sem qualquer outro tipo de informação.

Sobre o terceiro botão:

É um botão apenas com a palavra "SALVAR". Esse exemplo é bom, prático e evitar confusões. Qualquer usuário poderá identificar o significado do botão. Porém, vale ressaltar que a "aparência" ou "estilo" de botão está definida apenas através de CSS e assim um leitor de telas o identificará como "link".

Sobre o quarto botão:

É um botão com informação visual e também literal. É uma boa opção para introduzir o conceito de ícone em botões com a palavra acompanhando nos botões, ainda mais quando temos a troca de ícone que significa o salvamento de informações (disquete pela nuvem). Além disso, há marcações de ARIA que identifica o elemento corretamente como um "botão" caso se utilize leitores de tela. É a opção mais completa e viável para utilizar, quando se usa um link com aparência de botão.

Sobre o quinto botão:

É um botão idêntico como o anterior no quesito visual, porém com a melhor forma de se implemetar o elemento "botão", do ponto de vista técnico. Foi utilizado a tag < button > ao invés de um link < a >, evitando assim a utilização de controles ARIA desnecessários.

Exemplo: Paginação

Modelo RUIM e POUCO ACESSÍVEL (original do MATERIALIZECSS)

  • chevron_left
  • 1
  • 2
  • 3
  • chevron_right

Por que este modelo pode ser considerado ruim e pouco acessível?

  • As marcações de "página anterior" e "próxima página" aparecem apenas com a utilização dos sinais de maior e menor (o que vai contra a diretriz 1.3.3);
  • A marcação visual da página atual está com pouco contraste;
  • Não há mudança visual clara quando utilizamos o mouse para selecionar os elementos;
  • A página atual (2) possui link. Não é necessário manter o link uma vez que é a página que o usuário está navegando atualmente;
  • Não há identificação clara no código indicando qual a página selecionada e quais os elementos para ir para a anterior e próxima.

Modelo seguindo BOAS PRÁTICAS

  • chevron_left página anterior
  • 1
  • 2
  • 3
  • próxima página chevron_right

Por que este modelo pode ser considerado bom e acessível?

  • As marcações de "página anterior" e "próxima página" estão bem definidas e não dependem apenas do visual;
  • O contraste demarcando a página atual é forte, auxiliando usuários com baixa visão;
  • Há mudança visual tanto com o uso do mouse (hover), quanto com o uso do teclado (focus);
  • A página atual (2) possui link com verbalização de "página atual" (também poderia não ter link, uma vez que é a página atual);
  • Há marcações ARIA e identificações que facilitam a leitura do conteúdo por tecnologias assistivas;
  • No exemplo não foi aplicado, mas para casos em que haja muitas páginas, também pode ser aplicado botões de "primeira página" e "última página";
  • Importante: A semântica pode ser melhorada ainda mais, construindo o elemento envolto da tag NAV.
