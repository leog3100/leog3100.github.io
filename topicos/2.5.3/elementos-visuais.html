Exemplos para a diretriz 2.5.3 - Rótulo no nome acessível

Os exemplos abaixo demonstram boas práticas para determinar o significado de elementos independentemente da tecnologia e recurso utilizado.

A fim de agilizar o processo de criação, utilizei o framework MATERIALIZECSS open_in_newlink externo para a construção dos modelos abaixo.

Exemplo: Botão - Formato

No exemplo abaixo temos 5 botões vermelhos, um ao lado do outro.

save cloud salvar cloudsalvar

Sobre o primeiro botão:

Trata-se de um botão redondo com o ícone de um disquete, com a ideia de representar um botão para salvar. Porém vale lembrar que o disquete fez parte de uma época que já se passou e atualmente os mais jovens podem não saber do que se trata este elemento, se for mantido apenas o ícone no botão sem qualquer outro tipo de informação.

Sobre o segundo botão:

Trata-se de um botão redondo com o ícone de uma nuvem, com a ideia de representar um botão para salvar. Algumas iniciativas tendem a substituir o ícone do disquete para o ícone de nuvem para representar o salvamento na "cloud" (observem o ícone do botão SALVAR do codepen no alto da página quando estiver no modo de visualização de código). Porém vale lembrar que nem todos os usuários estão acostumados com essa identificação e podem não saber do que se trata se o botão for mantido apenas com o ícone sem qualquer outro tipo de informação.

Sobre o terceiro botão:

Trata-se de um botão apenas com a literal "SALVAR". É um bom exemplo, prático e direto para se utilizar. Qualquer usuário saberá identificar claramente para que serve o botão. Porém, vale ressaltar que a "aparência" de botão está definida apenas através de CSS e neste caso um leitor de telas lerá o elemento como um "link".

Sobre o quarto botão:

Trata-se de um botão com informação visual e também literal. É uma boa opção para fazer com que os usuários se acostumem com elementos visuais em seu site, ainda mais se tratando de uma troca de ícone que significa o salvamento de informações (disquete pela nuvem). Além disso, há marcações de ARIA identificando o elemento corretamente como um "botão" caso se utulize leitores de tela. É a opção mais completa para se utilizar, quando se usa elemento de link com aparência de botão.

Sobre o quinto botão:

É um botão exatamente como o anterior, porém com a melhor forma de se construir o elemento do ponto de vista técnico. Aqui utilizamos a tag < button > ao invés de um link < a >, além disso, evitamos dessa forma a utilização de controles ARIA de forma desnecessária.

Exemplo: Paginação

Modelo RUIM e POUCO ACESSÍVEL (original do MATERIALIZECSS)

  • chevron_left
  • 1
  • 2
  • 3
  • chevron_right

Por que este modelo pode ser considerado ruim e pouco acessível?

  • As marcações de "página anterior" e "próxima página" aparecem apenas com a utilização dos sinais de maior e menor (o que vai contra a diretriz 1.3.3);
  • A marcação visual da página atual está com pouco contraste;
  • Não há mudança visual clara quando utilizamos o mouse para selecionar os elementos;
  • A página atual (2) possui link. Não é necessário manter o link uma vez que é a página que o usuário está navegando atualmente;
  • Não há identificação clara no código indicando qual a página selecionada e quais os elementos para ir para a anterior e próxima.

Modelo seguindo BOAS PRÁTICAS

  • chevron_left página anterior
  • 1
  • 2
  • 3
  • próxima página chevron_right

Por que este modelo pode ser considerado bom e acessível?

  • As marcações de "página anterior" e "próxima página" estão bem definidas e não dependem apenas do visual;
  • O contraste demarcando a página atual é forte, auxiliando usuários com baixa visão;
  • Há mudança visual tanto com o uso do mouse (hover), quanto com o uso do teclado (focus);
  • A página atual (2) possui link com verbalização de "página atual" (também poderia não ter link, uma vez que é a página atual);
  • Há marcações ARIA e identificações que facilitam a leitura do conteúdo por tecnologias assistivas;
  • No exemplo não foi aplicado, mas para casos em que haja muitas páginas, também pode ser aplicado botões de "primeira página" e "última página";
  • Importante: A semântica pode ser melhorada ainda mais, construindo o elemento envolto da tag NAV.
